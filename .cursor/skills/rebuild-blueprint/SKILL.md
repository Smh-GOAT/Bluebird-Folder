# 重建蓝图工作流

一条"固定工作流"（照这个来就很稳）

## 角色分工

- **Kimi-K2.5**：输出《重建蓝图》
- **GPT-5.3-Codex**：按蓝图落地
- **循环**：做完一个模块回 Kimi 审查

---

## 阶段一：输出《重建蓝图》（Kimi-K2.5 负责）

### 输入要求
用户提供：现有项目路径或代码库

### 分析步骤

1. **读取项目结构**
   - 列出根目录文件
   - 识别技术栈（package.json, requirements.txt, Cargo.toml 等）
   - 分析现有架构

2. **输出《重建蓝图》**

使用模板：[templates/blueprint-template.md](templates/blueprint-template.md)

包含以下内容：
- 技术栈建议（按本地部署环境）
- 目录结构
- MVP 路线（先跑起来的最小功能链路）
- 模块拆分 + 每个模块的接口契约（请求/响应/错误码）+ JSON schema

---

## 阶段二：按蓝图落地（GPT-5.3-Codex 负责）

### 工作模式

每次只做一个**可运行增量**：

```
1. 改哪些文件 → 明确列出文件路径
2. 完整代码 → 直接可运行的代码
3. 运行命令 → 如何验证能启动
```

### 出错处理流程

```
用户贴日志 → AI 分析错误 → 迭代修复 → 直到能启动
     ↑___________________________________________|
```

**原则**：修到能启动再下一步，禁止跳过问题

---

## 阶段三：审查（回到 Kimi-K2.5）

每完成一个模块，执行检查清单：

- [ ] **是否偏离蓝图？**
  - 对比原设计与实现
  - 记录偏差及原因

- [ ] **是否遗漏边界情况？**
  - 空值处理
  - 异常流程
  - 并发安全

- [ ] **是否需要抽公共层？**
  - 重复代码识别
  - 工具函数提取
  - 配置集中管理

### 审查后决策

| 结果 | 动作 |
|------|------|
| 通过 | 进入下一个模块 |
| 需调整 | 返回阶段二修改 |
| 需重新设计 | 返回阶段一更新蓝图 |

---

## 使用方式

### 触发词
- "按重建蓝图工作流来"
- "开始重建"
- "输出蓝图"
- "按工作流来"

### 执行流程

```
用户触发 → 阶段一（Kimi 输出蓝图）→ 用户确认
                                    ↓
                        阶段二（Codex 落地模块1）
                                    ↓
                        阶段三（Kimi 审查模块1）
                                    ↓
                        通过？→ 阶段二（模块2）...
                                    ↓
                              全部完成
```

---

## 检查清单模板

每个阶段完成后复制此清单跟踪进度：

```markdown
## 当前进度

- [ ] 阶段一：《重建蓝图》输出完成
- [ ] 阶段二：模块1 落地完成
- [ ] 阶段三：模块1 审查通过
- [ ] 阶段二：模块2 落地完成
- [ ] 阶段三：模块2 审查通过
- [ ] ...
```
